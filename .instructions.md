# AI Agent Instructions for OmniTools MCP Server

These instructions guide AI agents working on this project. Follow these rules to maintain consistency and quality.

---

## Core Principles

### 1. The Plan is Immutable
- **`docs/plans/initial-plan.md` is the source of truth.** Do NOT modify it.
- All technical decisions are locked in. If you need to deviate, consult the human first.
- `docs/plans/clarifications-needed.md` contains the decision rationale.

### 2. Document Everything
- Log all issues, bugs, and solutions in `/docs/ISSUES_LOG.md`
- Update `docs/CHANGELOG.md` for any user-facing changes
- Keep `README.md` current with setup instructions

### 3. Configuration Over Code
- All tunable parameters MUST be in `.env.example` with defaults
- Never hardcode timeouts, file paths, limits, or thresholds
- Environment variables follow this naming: `TOOL_TIMEOUT`, `MAX_FILE_SIZE`, `SEARCH_RESULT_LIMIT`, `ALLOWED_DIRECTORIES`

---

## Coding Standards

### TypeScript
- **Strict mode enabled.** No `any` types unless absolutely necessary.
- Use `zod` for ALL input validation (tool parameters, config, file paths)
- Prefer functional patterns over classes where appropriate
- Export types/interfaces for all public APIs

### Error Handling
- **User-facing errors:** Return clear, actionable messages to the LLM
  - Good: `"File not found: /path/to/file.txt. Ensure the path is correct and the file exists in an allowed directory."`
  - Bad: `"ENOENT error"`
- **Log everything:** Full stack traces + context to `logs/mcp-server.log`
- **Partial failures:** Report what succeeded and what failed individually
  ```typescript
  { 
    processed: 3, 
    failed: 1, 
    errors: [{ file: "doc.pdf", error: "Corrupted header" }] 
  }
  ```

### File Operations
- **Security first:** ALL file paths must be validated against `ALLOWED_DIRECTORIES` whitelist
- Sanitize paths to prevent traversal attacks
- Use absolute paths internally; resolve relative paths from user input
- Check file sizes before processing (respect `MAX_FILE_SIZE`)

### Tool Implementation
- Each tool wrapper in `src/tools/` must:
  1. Define Zod schema for inputs
  2. Validate file paths against whitelist
  3. Respect timeout settings (`TOOL_TIMEOUT`)
  4. Return structured results (never throw to LLM)
  5. Log detailed errors to file

## Testing Requirements

### When Adding New Tools
1. Write unit test in `/tests/unit/` for the tool wrapper
2. Add integration test in `/tests/integration/` for end-to-end flow
3. Update regression suite if behavior changes
4. Document example usage in `/examples/`

### Before Committing
- Run full test suite: `npm test`
- Check linting: `npm run lint`
- Verify build: `npm run build`
- Update documentation (README, CHANGELOG, etc.) if needed
- Test with LM Studio in VSCode (Phase 5+)

**Note:** Documentation only needs to be updated before committing, not after every code change.

## Project Structure Rules

### Where Things Go
- **Source code:** `/src/` (TypeScript only)
- **Tests:** `/tests/` (unit and integration separated)
- **Examples:** `/examples/` (sample MCP calls, usage docs)
- **Configuration:** `/config/` (schemas, not runtime values)
- **Logs:** `/logs/` (created at runtime, git-ignored)
- **Build output:** `/dist/` (git-ignored)

### What NOT to Commit
- `.env` (runtime config with secrets)
- `/logs/` (runtime logs)
- `/dist/` (compiled output)
- `/node_modules/` (dependencies)
- Any credentials, API keys, or personal data

## Development Workflow

### Phase-Based Development
Follow the roadmap in `initial-plan.md`:
1. **Phase 1:** Base server skeleton + MCP SDK integration
2. **Phase 2:** 10-15 core tools (text, document, basic utilities)
3. **Phase 3:** FFmpeg media tools
4. **Phase 4:** Dispatcher (search + run pattern)
5. **Phase 5:** Multi-platform testing

**Do not skip phases.** Each must be complete before moving forward.

### Issue Tracking
When you encounter a bug or problem:
1. Document it in `docs/ISSUES_LOG.md` immediately
2. Use the format: `[ISSUE-XXX]` with sequential numbering
3. Include: problem, context, solution, and prevention steps
4. Mark status: Open → In Progress → Resolved

### Feature Planning
When you discover a potential feature/improvement during development:
1. Create a new file in `docs/plans/planned/` named `feature-name.md`
2. One document per feature (keep it concise initially)
3. Include just enough information to start planning later
4. Don't research or flesh out the feature fully when first identified
5. Before implementing: fully detail the feature plan with requirements, architecture, and acceptance criteria
6. After implementation is complete, move the feature document from `docs/plans/planned/` to `docs/plans/complete/`

**Feature Document Template:**
```markdown
# Feature: [Name]

## Brief Description
One-paragraph overview of what this feature does.

## Motivation
Why is this needed? What problem does it solve?

## Initial Thoughts
- High-level approach
- Potential challenges
- Related components

## Status
Idea | Planning | Ready for Implementation | In Progress | Complete
```

### Git Submodule Management
- omni-tools lives at `/src/lib/omni-tools` as a git submodule
- Check version on server startup
- Prompt user if updates are available
- Never auto-update without user consent

### Git Workflow

#### Branching Strategy
- **main:** Production-ready code only
- **dev:** Integration branch for active development
- **feature/[name]:** New features (branch from `main`)
- **bugfix/[name]:** Bug fixes (branch from `main`)
- **hotfix/[name]:** Urgent fixes (branch from `main`)

#### Mandatory Branch Sync Before New Work
Before creating any new branch, update local `main` and `dev`:

```bash
git checkout main
git pull origin main
git checkout dev
git pull origin dev
```

Never create a feature/bugfix branch from stale local refs.

**Branch Naming:**
- Use lowercase with hyphens: `feature/dispatcher-search`, `bugfix/timeout-handling`
- Be descriptive but concise

#### Commit Conventions
Follow Conventional Commits specification:

- **feat:** New feature or functionality
  - `feat: add omni_search tool with keyword ranking`
- **fix:** Bug fix
  - `fix: correct file path validation in whitelist check`
- **chore:** Maintenance tasks (dependencies, config, build)
  - `chore: update dependencies to latest versions`
- **docs:** Documentation only
  - `docs: update README with installation steps`
- **test:** Adding or updating tests
  - `test: add unit tests for text tool wrappers`
- **refactor:** Code restructuring without changing behavior
  - `refactor: extract path validation to utility function`
- **perf:** Performance improvements
  - `perf: optimize search ranking algorithm`
- **style:** Code style/formatting (no functional changes)
  - `style: apply prettier formatting to server.ts`

**Commit Message Format:**
```
<type>: <short summary> (max 50 chars)

<optional detailed description>
- What changed
- Why it changed
- Any breaking changes or important notes
```

#### When to Branch
- **Start any new work:** Create a feature/bugfix branch immediately
- **Multiple tasks:** Keep each feature/fix in its own branch
- **Experiments:** Always use a branch; never experiment on `dev` or `main`
- **Branch source:** All non-release work branches MUST start from `main`

#### When to Commit
- **Logical units of work:** One commit per complete, testable change
- **Working state:** Code compiles and tests pass (or skip with `--no-verify` if intentional WIP)
- **Documentation updates:** Only commit docs when ready to commit code
- **Before switching branches:** Commit or stash your changes

**Commit Frequency Guidelines:**
- Too frequent: Every line change
- Good: Complete implementation of a single function/module
- Too infrequent: 1000+ lines touching multiple features

#### When to Merge
- **Feature complete:** All acceptance criteria met
- **Tests passing:** Full test suite passes locally
- **Code reviewed:** If working with others, get review first
- **Documentation updated:** README, CHANGELOG, and relevant docs current
- **No conflicts:** Rebase or merge latest `main` into your branch first

**Merge Process:**
```bash
# Update your branch with latest main
git checkout main
git pull origin main
git checkout feature/your-feature
git rebase main  # or merge main if you prefer

# Open PR: feature/your-feature -> dev
# Do not merge directly to dev from local git commands.

# Release path
# Open PR: dev -> main
# After PR merge to main, create/publish release.

# Clean up
git branch -d feature/your-feature
```

#### When to Push
- **End of work session:** Push branches to back up your work
- **Before asking for help:** Push so others can see your code
- **Protected branches:** Never push directly to `dev` or `main`; use PRs
- **Main branch:** Changes only via approved PR from `dev`

#### When to Pull
- **Start of work session:** Pull latest changes before starting
- **Before creating a branch:** Ensure both `main` and `dev` are updated locally
- **Before merging:** Ensure you have latest `main` and `dev` before opening PR
- **Conflicts reported:** Pull to get the conflicting changes
- **Regularly:** Pull `main` and `dev` periodically to stay in sync (daily if active development)

## Version Control & Maintenance

### .gitignore Management
- **Location:** Root directory (`.gitignore`)
- **Keep it clean and legible:** Organized sections with comments
- **Use wildcards liberally:** `*.log` instead of listing individual files
- **Standard sections:**
  - Dependencies (`node_modules/`)
  - Build output (`dist/`, `build/`)
  - Environment & secrets (`.env`)
  - Logs (`logs/`, `*.log`)

## Security & Performance

### Security Checklist
- [ ] File paths validated against whitelist
- [ ] Input sanitized (no path traversal: `../`, absolute paths checked)
- [ ] Subprocesses spawned with minimal permissions
- [ ] No sensitive data logged to user-facing errors
- [ ] Timeouts enforced to prevent DOS

### Performance Checklist
- [ ] Tools respect `TOOL_TIMEOUT` (default 60s)
- [ ] File size checked before processing (`MAX_FILE_SIZE`)
- [ ] Large operations use streaming when possible
- [ ] Search results limited (default 10, configurable)

## Configuration Reference

### Required Environment Variables
Define these in `.env.example` with defaults:

```bash
# Tool execution
TOOL_TIMEOUT=60                          # seconds
MAX_FILE_SIZE=52428800                    # 50 MB in bytes

# Dispatcher
SEARCH_RESULT_LIMIT=10
SEARCH_RANKING_METHOD=keyword             # keyword | semantic (future)

# Security
ALLOWED_DIRECTORIES=/home/user/workspace  # comma-separated

# Logging
LOG_LEVEL=info                            # debug | info | warn | error
LOG_FILE=logs/mcp-server.log
```
